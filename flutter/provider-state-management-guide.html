Day 16: Provider State Management (Basics & ChangeNotifier)

Learn to manage global state cleanly and efficiently across your app using the Provider package. üß†

üß† Why Do We Need Provider?

Yesterday, we explored local state management using `setState` and callbacks. But as apps grow, passing data through widget constructors (known as **prop drilling**) becomes messy and hard to maintain. **Provider** solves this by making data available anywhere in the widget tree, from a single source.

Imagine a user's theme preference. Multiple widgets across the app need this information. Without a state management solution, you would have to pass it down manually to every single widget, which is inefficient and leads to unreadable code.

üì¶ Adding Provider to Project

First, add the **provider** package to your `pubspec.yaml` file:

pubspec.yaml 

dependencies:
  flutter:
    sdk: flutter
  provider: ^6.0.5

Then, run `flutter pub get` in your terminal to install the dependency.

üìå Understanding ChangeNotifier

A **ChangeNotifier** is a simple class that holds state and provides a notification system. When data changes, you call `notifyListeners()`, which automatically tells any listening widgets to rebuild themselves with the new data.

Example: Counter with Provider

This is a practical example of how to implement a counter app using Provider.

1. Create a State Class

counter_provider.dart 

import 'package:flutter/foundation.dart';

class CounterProvider with ChangeNotifier {
  int _count = 0;

  int get count => _count;

  void increment() {
    _count++;
    notifyListeners(); // tells UI to rebuild
  }

  void decrement() {
    _count--;
    notifyListeners();
  }
}

2. Provide It at App Root

Make your `CounterProvider` available to all widgets by wrapping your `MaterialApp` with a `ChangeNotifierProvider`.

main.dart 

import 'package:flutter/material.dart';
import 'package:provider/provider.dart';
import 'counter_provider.dart';
import 'home_page.dart';

void main() {
  runApp(
    ChangeNotifierProvider(
      create: (context) => CounterProvider(),
      child: MyApp(),
    ),
  );
}

class MyApp extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return MaterialApp(
      home: HomePage(),
    );
  }
}

3. Consume It in Widgets

Now, any widget below the `ChangeNotifierProvider` can access the state. Use `Provider.of(context)` to read the state and rebuild the UI.

home_page.dart 

import 'package:flutter/material.dart';
import 'package:provider/provider.dart';
import 'counter_provider.dart';

class HomePage extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    final counter = Provider.of(context);

    return Scaffold(
      appBar: AppBar(title: Text("Provider Example")),
      body: Center(
        child: Text(
          "Count: ${counter.count}",
          style: TextStyle(fontSize: 30),
        ),
      ),
      floatingActionButton: Row(
        mainAxisAlignment: MainAxisAlignment.end,
        children: [
          FloatingActionButton(
            onPressed: counter.increment,
            child: Icon(Icons.add),
          ),
          SizedBox(width: 10),
          FloatingActionButton(
            onPressed: counter.decrement,
            child: Icon(Icons.remove),
          ),
        ],
      ),
    );
  }
}

üëâ Now any widget in the app can access `CounterProvider` without passing props around. You can also use the `Consumer` widget to rebuild only specific parts of the widget tree for better performance.

üîÑ Provider Widget Types

Flutter's Provider package offers different types of widgets for different use cases:

  • **Provider**: For simple, static values that don't change.
  • **ChangeNotifierProvider**: The most common, used for mutable state that can be updated.
  • **MultiProvider**: A convenient way to provide multiple providers at once.
  • **Consumer**: A widget that only rebuilds when the state it listens to changes, improving performance.

üéØ When to Use Provider?

‚úÖ **Use Provider when:**

  • State needs to be shared **globally** or across a large part of the app.
  • Data updates affect multiple widgets that are far apart in the widget tree.

Common examples include user authentication status, items in a shopping cart, or the app's theme settings.

‚ùå **Don‚Äôt use Provider for:**

  • Simple one-off local state (stick with `setState`).

‚úÖ Checklist for Today

  • Installed Provider package
  • Created `ChangeNotifier` class
  • Used `ChangeNotifierProvider` at the app root
  • Accessed data with `Provider.of` or `Consumer`
