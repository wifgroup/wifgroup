Day 15: Local State Management in Flutter

Learn how to make your apps dynamic and interactive by managing state with setState and callbacks. üß†

üß† What is State?

In Flutter, **state** is any data that changes over time in your app. When this data changes, the UI should rebuild to reflect the new state. If you have a counter, a toggle switch, or a text input, you are dealing with state.

  • A counter value üßÆ
  • A toggle switch (on/off) üîò
  • A text input value üí¨

üìå Using `setState()`

The simplest way to update state in a **StatefulWidget** is by using `setState()`. This function tells Flutter that some data has changed, and it should rebuild the widget to reflect the new state.

Example: Counter App

Counter App Example 

import 'package:flutter/material.dart';

class CounterApp extends StatefulWidget {
  @override
  _CounterAppState createState() => _CounterAppState();
}

class _CounterAppState extends State {
  int _count = 0;

  void _increment() {
    setState(() {
      _count++; // update state
    });
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(title: Text("Local State - setState")),
      body: Center(
        child: Text(
          "Count: $_count",
          style: TextStyle(fontSize: 30),
        ),
      ),
      floatingActionButton: FloatingActionButton(
        onPressed: _increment,
        child: Icon(Icons.add),
      ),
    );
  }
}

üëâ Every time you press the button, `setState()` rebuilds the widget with the new count.

üìå Passing Data with Callbacks

In many cases, a child widget needs to update the state of its parent. You can achieve this by passing a function (a **callback**) from the parent to the child. When the child widget is interacted with, it calls the function, which then triggers a `setState` in the parent.

Example: Child Button Updates Parent State

Callback Example 

class ParentWidget extends StatefulWidget {
  @override
  _ParentWidgetState createState() => _ParentWidgetState();
}

class _ParentWidgetState extends State {
  String message = "Hello üëã";

  void updateMessage(String newMessage) {
    setState(() {
      message = newMessage;
    });
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(title: Text("Callbacks Example")),
      body: Column(
        mainAxisAlignment: MainAxisAlignment.center,
        children: [
          Text(message, style: TextStyle(fontSize: 24)),
          SizedBox(height: 20),
          ChildButton(onPressed: updateMessage),
        ],
      ),
    );
  }
}

class ChildButton extends StatelessWidget {
  final Function(String) onPressed;

  ChildButton({required this.onPressed});

  @override
  Widget build(BuildContext context) {
    return ElevatedButton(
      onPressed: () => onPressed("You clicked the child button! üéâ"),
      child: Text("Click Me"),
    );
  }
}

üëâ Here, the child widget communicates back to the parent using a callback.

üéØ When to Use Local State?

Use `setState` and callbacks when:

  • The state is **local to one widget**.
  • The state **doesn‚Äôt need to be shared widely** across the app.

Examples include form inputs, counters, toggles, and simple animations. For larger apps, it's better to use a dedicated state management solution (like Provider, Riverpod, or Bloc), which we will explore later.
